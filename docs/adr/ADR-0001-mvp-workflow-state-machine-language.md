# ADR-0001: MVP阶段的流程收敛与任务状态机语言选择

- Status: Proposed
- Date: 2026-04-13
- Deciders: Project Owner + Engineering
- Related: `# Project Context & Tech Stack Selection.md`

## 背景
当前项目目标是“电商视觉生成工作流”，但业务流程尚未完全固化。
此阶段如果先追求语言升级（例如直接上 Rust），会增加系统复杂度和交付风险。

## 要解决的问题
1. 业务流程不完整时，如何继续推进研发而不返工。
2. 任务状态机应该用 Python 还是 Rust 实现。
3. 如何把该决策变成可执行的工程计划。

## 决策
1. **流程策略（先收敛再扩展）**
- 先冻结一个 `MVP业务闭环`：`提交生成请求 -> 排队 -> 调用ComfyUI -> 回调/轮询 -> 产物入库 -> 前端可查询状态`。
- 不等待全流程澄清才开发；采用“里程碑门禁”方式，每周补齐并锁定新增流程。

2. **状态机实现语言：MVP阶段采用 Python（FastAPI + Redis + ARQ/Celery）**
- 状态机核心是“状态定义、迁移约束、幂等与重试”，不是 CPU 密集计算。
- Python 在当前栈内实现成本最低、调试效率最高，与 FastAPI/Redis/ComfyUI 集成最短路径。
- 通过显式迁移表和事件模型，Python 可以达到足够工程可靠性。

3. **Rust 引入策略：按指标触发，不预设提前引入**
- 仅当满足以下任一条件时，再引入 Rust 服务或 Rust 扩展：
  - 状态机吞吐稳定超过 `1k+ transitions/s` 且 Python 成本显著上升。
  - 单机 CPU 使用率长期高于 `70%` 且优化后仍无法满足 SLO。
  - 出现 Python 侧难以规避的并发一致性/尾延迟问题。
- Rust 作为“性能优化阶段工具”，而不是“问题定义阶段工具”。

4. **队列默认建议：MVP优先 ARQ（可替换）**
- 原因：原生 async，和 FastAPI 协作简单，学习与运维成本低。
- 保留抽象层（`TaskDispatcher` 接口），后续可切换 Celery 而不改业务层。

## 状态机规范（v0）

### 状态
`queued -> running -> postprocessing -> succeeded | failed | timeout | cancelled`

### 关键约束
- 同一 `job_id` 只能有一个终态（`succeeded/failed/timeout/cancelled`）。
- 使用 `idempotency_key = hash(user_id + input + workflow_version)` 防止重复任务。
- 所有外部回调必须携带 `request_id/job_id`，缺失即拒绝。
- 重试采用指数退避，达到上限后转 `failed`。

### 最小事件集
- `job_submitted`
- `worker_started`
- `comfyui_accepted`
- `comfyui_progress`
- `comfyui_completed`
- `postprocess_completed`
- `job_failed`
- `job_timed_out`
- `job_cancelled`

## 备选方案与取舍

### 方案A：立即用 Rust 重写状态机
- 优点：类型与并发控制更强，理论性能更高。
- 缺点：与当前 Python 主栈割裂，开发/调试/交付周期显著增加。
- 结论：当前阶段不采用。

### 方案B：仅 Python + 内存状态（不落 Redis）
- 优点：实现极快。
- 缺点：进程重启即丢状态，不满足生产可用性。
- 结论：不采用。

### 方案C：Python + Redis 持久化状态机（本决策）
- 优点：实现速度与可靠性平衡最佳，符合当前团队能力与项目节奏。
- 缺点：极限吞吐不如 Rust，需要后续指标驱动优化。

## 影响

### 正向
- 本周即可开始编码，不再被“流程必须一次性完整”阻塞。
- 降低前期技术复杂度，优先验证业务价值和生成链路可用性。

### 负向
- 后续若规模快速增长，可能需要状态层重构或局部 Rust 化。
- 需要严格执行观测与压测，否则无法判断何时升级。

## 实施计划（两周）
1. 第1-2天：固化 `MVP业务闭环`、补齐事件字典与错误码。
2. 第3-5天：实现状态机与任务接口（含幂等、超时、重试）。
3. 第6-8天：接入 ComfyUI 回调/轮询，完成端到端链路。
4. 第9-10天：压测与观测（成功率、P95 时延、重试率、成本/任务）。
5. 第11-14天：按指标复盘，决定是否进入下一阶段优化（含是否需要 Rust）。

## 验收标准
- 端到端成功率 >= 95%（可配置工作流样本集）。
- 重复提交不产生重复终态任务。
- 任务状态可追踪且可恢复（服务重启后不丢状态）。
- 可以输出至少 1 份性能与成本周报。

## 待确认事项
- 视觉生成里程碑顺序：`模特替换`、`背景替换`、`细节保真`、`批量生成` 的优先级。
- 产物存储方案（对象存储/本地）与保留周期。
- 失败重试上限和超时策略的业务容忍度。
